# 多设备同步规范

## 1. 目标

建立一套可执行的多设备工作方式：

- 当前 Mac 作为主工作机
- 一台 PC 作为辅助机
- GitHub 作为统一同步中枢
- Codex 本机技能目录作为本机运行依赖

目标不是让所有设备完全实时一致，而是让“规则一致、内容可追溯、冲突可控、恢复可重建”。

## 2. 基本原则

1. 主机优先：所有新规则先在 Mac 上确认。
2. 仓库优先：长期知识、规则、项目骨架以 GitHub 仓库为准。
3. 设备分工：PC 只做执行端，不做规则源头。
4. 明确边界：临时文件、缓存、系统状态不纳入同步范围。
5. 单向发布：推荐 Mac -> GitHub -> PC，不建议双向自动覆盖。

## 3. 同步分层

### 3.1 必须同步

以下内容必须进入 GitHub，并在 Mac 与 PC 之间保持一致：

- `Agent.md`
- `全局工作台.md`
- `全局复利与踩坑日志.md`
- `新项目SOP.md`
- 项目文档
- 技能文档和技能入口索引
- 关键脚本

### 3.2 允许不同步

以下内容可以只留在本机，不作为同步目标：

- 编辑器缓存
- 临时日志
- 下载文件
- 构建产物
- 系统级状态
- 自动生成中间文件

### 3.3 只在确认后同步

以下内容必须人工确认后再同步：

- 草稿
- 未完成方案
- 临时测试文件
- 调试结论
- 可能影响目录结构的改动

## 4. 单向同步流程

### 标准流程

1. 在 Mac 上修改规则、文档或 skill 入口。
2. 先在本机确认可读、可用、可恢复。
3. 提交到 GitHub 仓库。
4. PC 只从 GitHub 拉取更新。
5. 若 PC 产生有效补充，回写 GitHub，再由 Mac 复核。

### 禁止流程

- 不允许 PC 与 Mac 互相盲目覆盖。
- 不允许把本地临时状态直接当作主版本。
- 不允许在两个设备上同时编辑同一份规则文档后不做合并。

## 5. 冲突处理规则

当出现内容不一致时，按以下顺序处理：

1. 先看 GitHub 仓库版本。
2. 再看 Mac 本机确认版本。
3. 最后看 PC 本机版本。

如果三者冲突：

- 以最近一次明确确认的版本为准
- 如果是新规则，先更新 `Agent.md` 和 `全局工作台.md`
- 如果只是单个项目特例，只更新项目目录内文件

## 6. 技能与本机目录

Codex 技能分两层：

- GitHub 仓库中的技能源码和说明
- 本机 `~/.codex/skills` 中的可调用安装目录

可执行规则：

1. 仓库是技能的标准来源。
2. Mac 的 `~/.codex/skills` 是当前运行环境的可用副本。
3. 要“随时单独调用”的 skill，必须在本机 `~/.codex/skills` 有入口。
4. 工作区可以保留软链接或索引，方便人眼查看，但不替代本机技能目录。

## 7. 项目结构要求

每个新项目必须以目录为边界：

```text
project-编号-项目名称/
  01-需求说明.md
  02-执行记录.md
  03-交付说明.md
  assets/
  notes/
  outputs/
```

约束：

- 项目相关内容只放在对应项目目录内。
- 不允许散落到电脑其他位置。
- 任务结束时必须做目录收口。

## 8. 恢复流程

新机器或新终端恢复时，按这个顺序：

1. 配置 Git 身份。
2. 配置 GitHub 登录。
3. 拉取仓库。
4. 安装需要的 Codex skills 到 `~/.codex/skills`。
5. 检查工作区是否有技能入口索引。
6. 检查 `Agent.md` 和三份全局文档是否存在。

## 9. 为什么这套方案可行

### 9.1 核心推理

这套方案把系统拆成三类对象：

- **规则源**：GitHub 仓库
- **执行环境**：Mac 和 PC
- **本机运行依赖**：`~/.codex/skills`

一旦角色分清，问题就从“多设备实时同步”变成“仓库发布 + 本机安装 + 目录索引”。

### 9.2 可行性依据

1. GitHub 适合存长期规则，因为它有版本历史、分支和审计。
2. 本机 `~/.codex/skills` 适合存运行副本，因为 Codex 读取技能时依赖本地目录。
3. 工作区软链接或索引适合做可见入口，因为它不复制逻辑，只暴露路径。
4. 单向发布减少冲突，因为只有一个主写入方向。

### 9.3 反证检查

如果采用双向自动同步，会出现：

- 两台机器同时改同一文档
- 临时文件误入仓库
- 旧配置覆盖新配置
- 不能判断哪台机器代表真相

这会让维护成本指数级上升，所以不采用。

### 9.4 结论

只要把“规则源”和“运行副本”分开，这套方案是可执行的。
其前提不是网络同步本身，而是：

- 仓库持续作为真相来源
- 本机技能目录保持安装
- 设备之间只做受控发布，不做无约束互写

## 10. 执行检查清单

每次改动后检查：

1. 规则是否已写入仓库。
2. Mac 本机是否已经能读到。
3. PC 是否只作为执行端。
4. `~/.codex/skills` 是否有对应 skill。
5. 工作区是否有清晰入口。
6. 是否有未确认的临时内容泄漏到仓库。
