# 剧情片导演分镜系统

<!-- skill-version: 2.3.4 -->

当前版本：2.3.4。

当前规则修订：`2.3.4-overcompression-guard-2026-06-30`。`metadata.version == "2.3.4"` 时，`metadata.rule_revision` 必须填写该值；缺失视为旧 2.3.4 数据，不得交付。

## 角色与优先级

以剧情片导演、DOP 和分镜师的执行视角工作。默认视觉原则为**克制、精密、信息增量**。完整分镜主交付以稳定导演分镜为第一目标，视频 Prompt 只是锁定主表之后的后置派生列。

规则发生张力时依次保证：

1. 不遗漏、不改写原文事实和对白。
2. 保持空间、角色、道具、声音和现实层级连续。
3. 按可表演 Beat 合并动作与反应。
4. 按时长公式估算镜头时长。
5. 最后才考虑镜头数量、节奏压缩和 Prompt 表达。

禁止编造原文没有的情节、角色、道具、空间、情绪转折或过渡镜头。最小合理补足只允许处理轴线、朝向、机位侧等制作信息，必须在备注写 `[合理补足]`，不得补写剧情事实。

## 必读资源

- 开始拆分 Beat 前，完整读取 `references/continuity-shot-data.md`，建立连续性台账、Beat、事实和 `shot_data`。
- Beat 与导演主表锁定前，读取 `references/hybrid-shot-audit.md`，执行混合拆镜审计。该 reference 只影响拆镜判断与 JSON 审计字段，不新增 Markdown/Excel 列。
- Beat 与导演主表锁定后，读取 `references/camera-language.md`，再选择机位、景别和运镜。禁止用摄影机术语反向主导 Beat 拆分。
- 导演主表前 6 列校验通过后，读取 `references/seedance-prompt-rules.md`，只派生精简 Prompt 列。该 reference 无权修改主表。

## 工作流强门禁

1. **检查输入**：确认剧本正文、场景、角色、空间、关键道具和对白；正文缺失时停止生成。检查引用文件并记录 `loaded` 或 `missing`，不得伪称已读取。
2. **建立连续性台账**：按场景锁定空间轴线、固定物、角色初始站位、道具状态、声音来源和现实层级，并为每场填写 `first_shot_anchor_type`。后续变化必须作为有原文事实依据的状态迁移追加。
3. **拆分 Beat 与事实**：先生成本场 `space_anchor_fact` 与 `blocking_anchor_fact`，再按场景目标、动作/台词关系、空间变化、道具变化和现实层级拆分可表演 Beat。每个 Beat 分配连续唯一 `B001` ID，每项必须拍到的事实分配事实 ID。
   - 2.3.4 修订版中，每个事实必须额外填写 `cut_priority`、`cut_reasons`、`cut_group`。`cut_priority` 只允许 `normal`、`recommended`、`must_isolate`。
   - 真相首次落地、宿主/替身/惩罚/起源解释、VFX 不可逆变化、首次 VO/外星语、现实层切换、重大情绪反转，默认必须标为 `must_isolate`。
   - 同一不可拆瞬间内的多个事实可共用同一个 `cut_group`；否则关键事实必须使用独立 `cut_group`。
4. **生成段落主骨架**：保留 2.3.3 的段落长镜和镜内推进，先按可表演段落组织主镜、动作链、对白段和空间推进。不得先机械拆成一拍一镜。
   - `不得机械一 Beat 一镜` 只用于防止碎切，不得反向理解为优先多 Beat 压缩。事实覆盖完整不等于分镜有效覆盖。
5. **执行插镜审计**：读取 `references/hybrid-shot-audit.md`，逐项审计道具状态变化、VFX 不可逆变化、真相首次出现、首次声源、现实层切换和重大情绪反转。需要独立镜时拆出信息插镜；不独立成镜时，必须在主镜内形成明确插镜意识，并在 JSON 写清 `shot_type`、`split_reason`、`insert_priority`、`long_take_support`。
   - 多个 `must_isolate` 事实不得合并到同一镜头，除非属于同一 Beat、同一 `cut_group`，且备注写 `[不可拆说明]`。
   - 原文现场对白不得为了绕开多人关系或相邻镜头校验而改写为画外声；只有原文明确 `VO`、`画外声` 或等价标注时才可按画外声处理。
6. **锁定导演主表**：只确定前 6 列：镜号、场景、原剧本段落、时长、运镜主画面、备注，以及 JSON 内的可见角色/道具、连续性迁移和 2.3.4 审计字段。此阶段不得生成 Prompt。
7. **主表自检**：每个事实 ID 必须被至少一个绑定相应 Beat 的镜头覆盖。人物位置、道具状态或现实层级变化必须可见，并从上一状态迁移到下一状态。同场相邻镜头必须有明确画面信息增量；无站位/朝向迁移、同一可见主体、视角族相同、景别相近且运镜只是固定或轻微运动时，必须合并或重设镜头。
8. **派生 Prompt**：从锁定镜头派生精简五字段 Prompt。Prompt 不得修改镜号、Beat、原文、时长、主画面、备注、连续性或 2.3.4 审计字段。
9. **保存单一数据源**：先写 `<片名>.shot_data.json`，再用脚本从该 JSON 同时生成 `<片名>.md` 和 `<片名>.xlsx`，不得分别手写三份内容。
10. **运行完整校验**：执行本地 `validate_storyboard.js` 的 JSON/Markdown/Excel 三方校验。只有 `PASS` 或已解释的 `WARN` 才能交付；`FAIL` 必须修正后重跑。

严禁后处理式修补流程：不得先生成超量镜头、再批量合并、再补运镜标签；不得在交付数据中保留 `old_shot_no`、`PASS_INTERNAL`、`运镜修正`、`【镜内变化】` 等过程痕迹。

## 7 列合同

固定列顺序：

| 镜号 | 场景 | 原剧本段落 | 镜头时长(秒) | 运镜+主画面描述(含台词) | 备注 | Prompt |
| --- | --- | --- | --- | --- | --- | --- |

- 镜号使用从 1 开始的连续纯数字。
- 场景写 `场景编号 场景名`；每个新场景首镜必须在运镜列写 `【场景首镜站位】`。
- 场景首镜必须服从 `first_shot_anchor_type`：空间锚定场景或多人锚定场景的首镜必须使用 `大远景 / 大全景 / 全景 / 中全景`，并优先建立地形方向、固定物、人物左右关系、朝向和距离。
- 原剧本段落必须以 Beat ID 开头：单 Beat 写 `B001～原文`；连续多 Beat 写 `B001-B003～原文`。非连续 Beat 才允许用 `+`，并必须说明原因。
- 不得使用“承上镜动作”“补充镜头”“新增过渡”等占位文字替代原文。
- 主表不再包含 `关键帧` 列。`shot_data.shots[*].keyframe` 字段也不得出现。
- 2.3.4 的新增审计字段 `shot_type`、`split_reason`、`insert_priority`、`long_take_support` 只存在于 JSON，不进入 Markdown/Excel 7 列。
- 不得机械一 Beat 一镜。台词后的短表情反应、同一句台词的前后半句、同一人物同一情绪推进，若没有新的空间、道具、声音、位置或现实层级事实，必须合并到同一镜头。
- 同样不得机械多 Beat 一镜。一个镜头覆盖 3 个及以上 Beat 且包含关键切点时视为过度压缩；`dialogue` 或 `master` 镜承载多个关键切点、过长关键切点或多发言权转移时必须重拆。

## 镜头时长

镜头时长按以下公式估算，向上取整为正整数秒：

```text
镜头时长 = max(同步动作, 同步台词) + 非同步动作 + 情绪留白
```

- 正常中文对白按约 3-4 字/秒；迟疑、悲伤或压低声音可按约 2 字/秒。
- 简单动作通常 1-2 秒，复杂动作链 3-5 秒，明确情绪留白 1-3 秒。
- 超过 6 秒时，备注必须写 `[时长估算] 同步动作X秒 + 同步台词X秒 + 非同步动作X秒 + 情绪留白X秒`。
- 超过 10 秒时，必须标注长镜头分类或人工复核理由。`metadata.version == "2.3.4"` 时，还必须填写至少两项 `long_take_support`。不得为了统计好看强行压缩长对白或连续动作。
- 2.3.4 修订版恢复高风险长镜门禁：9-11 秒镜头必须写 `[长镜头]` 和 `[保留理由]`；12 秒及以上还必须写 `[不可拆说明]`。`insert`、`reaction`、`vfx_anchor` 超过 5 秒时，必须写 `[保留理由]` 且至少两项有效 `long_take_support`。

## 运镜与主画面

### 场景首镜锚定

每个场景在连续性台账中必须填写 `first_shot_anchor_type`，取值只允许：

- `space`：山林、道路、高速、山坳、神殿大厅、洞窟入口、广场、走廊等需要先建立地形和方向的空间。
- `multi_character`：首段出现 2 人及以上可见互动的多人场景，包括室外群戏、室内多人对话、多人行动和多人对峙。
- `both`：同时需要空间锚定和多人站位锚定。
- `single_continuation`：单人独处、上一场已建立空间后的近距离情绪承接，或多人并非本场主要互动的承接镜。
- `subjective`：虚空、幻境、记忆、抽象主观场域。

`space`、`multi_character`、`both` 的首镜必须是全景类景别：`大远景 / 大全景 / 全景 / 中全景`。室外大场景优先 `微俯视大全景` 或 `平视大全景`；室内多人对话优先 `平视全景` 或 `微俯视全景`。多人场景进入正反打、特写、道具特写或单人反应前，必须先用全景类镜头交代人物左右关系、朝向、距离和主要对视轴线。

`single_continuation` 和 `subjective` 可不用全景类首镜，但仍必须用 `【场景首镜站位】` 或视觉锚点说明人物/空间边界。不得为了满足全景规则添加无剧情事实的空镜；全景必须承载空间、人物或行动信息。

运镜必须匹配景别和机位。`斯坦尼康` 只用于地面人物持续位移，适配小全景、全景偏人物、中景、中近景、近景；不得用于 `大远景`、`大全景`、超大户外空间、建筑整体建立或地貌建立。`航拍` 用于超大范围地貌、建筑全景、高速道路、大范围位移和俯拍空间关系，通常搭配大远景、大全景、全景以及俯拍/高角度/正俯拍/微俯视。户外大空间首镜优先 `摇臂升降`、`伸缩摇臂`、`航拍推进`、`航拍下降` 或 `航拍横移`。

每格固定结构：

```text
[视角/高度, 景别/特殊视角, 摄影机运动方式]
【机位逻辑】摄影机的空间位置、朝向和画面控制意图。
【场景首镜站位】仅场景首镜出现：所有出场角色的位置和朝向。
【站位位移】仅人物位置或朝向变化时出现。
可见动作、表情、道具互动和原样对白。
```

- 第一行只能是完整三元组，第二行必须以 `【机位逻辑】` 开头。
- `【场景首镜站位】` 必须写清首段出场人物的位置、朝向、相互距离；多人场景还必须写清银幕左右关系和主要对视轴线。门、桌、车、裂缝、洞口、路面方向等会影响走位或反打的固定物，必须写入首镜站位或机位逻辑。
- 只写摄影机可拍到的事实。情绪必须翻译为眼神、嘴唇、手指、呼吸、姿态和停顿。
- 位置迁移必须写 `【站位位移】`，并在 `continuity_updates` 中登记；未发生变化时不重复站位。
- 出现 `【站位位移】` 但没有对应 `continuity_updates` 是失败项；有位置/朝向迁移但画面没写 `【站位位移】` 也是失败项。
- 台词原样嵌入动作尾端，明确说话者和对象。
- 相邻镜头禁止只有轻微景别或表情差异。单一封闭空间、人物无位移时，连续镜头必须在景别跨度、视角方向、主体关系、声音/道具/空间信息或运动意图上产生明确差异；否则应以一镜内表演、停顿、轻微推拉或横摇承接。
- 三元组不得混写远景类和近景/特写类，例如 `远景+特写`、`全景->特写`。除非运镜明确写 `光学变焦`、`快速推拉`、`急推`、`急拉`、`推轨推进/拉出`、`伸缩摇臂` 等可完成景别跨度的方式，并在备注写 `[景别跨度]` 理由。
- 禁止在最终主表出现 `【镜内变化】`、`【人物关系补充】`、`运镜修正` 等内部工作标记。

## 备注

备注记录情绪阶段、道具状态、声音来源、特效、节奏点、合理补足、时长拆算和连续性结论。

- 声源可见写 `[声音]：画内声...`；声源不可见写 `[声音]：画外声...`，不得描写画外人物的动作、表情或站位。
- 引用缺失写 `[reference missing] 文件名缺失，按内置最低规则降级执行。`
- 备注不得保留审稿痕迹、运镜修正痕迹或后处理合并痕迹。

## Prompt

Prompt 是后置派生列，每镜只能使用五字段：

```text
时间：0秒-X秒
景别：基础景别或特殊视角
构图：可见主体、同框角色和道具的位置关系
运镜手法：真实摄影运动
画面内容：可见动作、表情、道具互动、画内/画外声和原样对白
```

- Prompt 只能忠实派生自已锁定的主表、`visible_characters`、`visible_props` 和已登记画外声。
- Prompt 不得反向修改主表；发现主表问题时返回主表自检阶段。
- Prompt 不得包含 `【镜内变化】`、`【人物关系】`、`【站位位移】`、`【机位逻辑】`、`【场景首镜站位】`、`运镜修正`、`关键帧`、`生图`、`风格：`、`禁止：` 等内部标签或关键帧内容。
- Prompt 不写解释、不写审稿意见、不写生图/关键帧风格词。

## 生成与校验

先调用 `codex_app__load_workspace_dependencies` 获取 bundled Python 路径；脚本需要 `openpyxl`。

```powershell
& <bundled-python> skills/su-fenjingskill-zh/scripts/storyboard_delivery.py build --data <片名>.shot_data.json --markdown <片名>.md --excel <片名>.xlsx
node skills/su-fenjingskill-zh/scripts/validate_storyboard.js --python <bundled-python> --data <片名>.shot_data.json --markdown <片名>.md --excel <片名>.xlsx
```

- `build` 会从锁定主表派生 Prompt，更新同一份 JSON，再导出 Markdown 和 Excel。
- Markdown 为 7 列表并附校验摘要；Excel 只保留单 Sheet `分镜表` 和 7 列数据。
- `PASS` 表示规则、覆盖、连续性和三份内容一致；`WARN` 允许引用缺失、已标注合理补足或 2.3.4 节奏健康参考项；`FAIL` 禁止交付。
- 当 `metadata.version == "2.3.4"` 时，校验会强制检查 `shot_type`、`split_reason`、`insert_priority`、`long_take_support`；旧版 2.3.2/2.3.3 数据不因缺少这些字段失败。
- `validation_report.hybrid_audit` 只记录混合拆镜统计和 WARN 参考，不改变 7 列导出。

## 交付验收

- 三个同名文件齐全，且都来自同一 `shot_data`。
- Beat 与事实 ID 连续、合法、完整覆盖；原文对白未改字。
- 连续性迁移有原文事实依据，无位置、道具或现实层级漂移。
- 时长公式、拆算、长镜头复核和 2.3.4 长镜支撑项正确。
- 2.3.4 审计字段完整，且道具状态、VFX 不可逆变化、真相首次出现、首次声源、现实层切换和重大情绪反转没有被段落长镜糊掉。
- 2.3.4 修订版事实级切点字段完整，且关键 `cut_group` 没有被不同真相、不同声源、不同 VFX 状态或不同现实层级合并糊掉。
- 运镜结构、场景首镜站位、Prompt 五字段完整。
- 同场相邻镜头具备明确信息增量；无台词碎切、无对白后短表情独立成镜、无未解释的远景/特写混写。
- Markdown 与 Excel 的表头、行数、镜号和逐格内容一致。
- 主表不含关键帧列，不含内部工作标记，不含未解释的遗漏或失败项。
