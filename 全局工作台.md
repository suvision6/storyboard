# 全局工作台

## 1. 目的

这个文档是所有项目的总入口，用来统一规范：

- code 的行为规则
- 命名规范
- 目录结构
- 文档使用规则
- 新项目启动流程
- 复用方法
- 踩坑记录的引用方式

## 2. 最高使用原则

1. 先理解真实目标，再开始执行。
2. 默认用户想要的是可执行结果，不是泛泛建议。
3. 如果用户纠正过我，最新纠正优先于旧假设。
4. 对不确定信息，必须标注来源：
   - `已验证`
   - `来自记忆`
   - `我的推断`
5. 任何时候都优先复用已有约定、历史决策、脚本、文档和记忆。

## 3. code 行为规则

1. 先查上下文，再动手。
2. 优先阅读：
   - `Agent.md`
   - `全局工作台.md`
   - `全局复利与踩坑日志.md`
   - `新项目SOP.md`
   - 当前项目内已有说明文件
3. 能复用的内容先复用，不能复用再新建。
4. 遇到不确定、冲突、风险较高的情况，先停下来校准。
5. 变更前先理解目录结构，避免把内容散落到电脑其他位置。
6. 所有新项目相关内容，必须收纳进对应项目文件夹。

## 4. 全局命名规范

### 4.1 文件夹命名

- 新项目文件夹统一格式：
  - `project-编号-项目名称`
- 示例：
  - `project-001-客户官网重构`
  - `project-002-数据看板优化`

### 4.2 文档命名

- 全局制度文档固定使用以下名称：
  - `Agent.md`
  - `全局工作台.md`
  - `全局复利与踩坑日志.md`
  - `新项目SOP.md`
- 项目内文档建议采用清晰前缀：
  - `01-需求说明.md`
  - `02-执行记录.md`
  - `03-交付说明.md`

## 5. 目录结构总原则

### 5.1 工作区总原则

所有相关内容必须放进对应项目文件夹，不能散落到电脑其他位置。

### 5.2 标准结构

```text
workspace/
  Agent.md
  全局工作台.md
  全局复利与踩坑日志.md
  新项目SOP.md
  project-001-项目名称/
    01-需求说明.md
    02-执行记录.md
    03-交付说明.md
    assets/
    notes/
    outputs/
```

### 5.3 目录职责

- `assets/`：图片、素材、原始附件
- `notes/`：过程记录、思考、草稿、结论
- `outputs/`：交付物、最终产物、可对外文件

## 6. 文档使用规则

### 6.1 全局工作台

- 作为总入口和总索引
- 汇总所有通用规则
- 记录命名规范、目录结构、工作方式
- 维护最新的跨项目共识

### 6.2 全局复利与踩坑日志

- 记录犯过的错误、踩过的坑、走过的弯路
- 记录未来可持续复用的方法
- 每次出现重复问题，先查这份文档

### 6.3 新项目SOP

- 规定每个新项目必须怎么创建、怎么归档、怎么命名
- 任何新项目开始前必须阅读

## 7. 项目启动总流程

1. 先读 `Agent.md`
2. 再读 `全局工作台.md`
3. 再读 `全局复利与踩坑日志.md`
4. 再读 `新项目SOP.md`
5. 创建 `project-编号-项目名称` 文件夹
6. 把所有相关内容全部放入该文件夹
7. 按需创建项目内子目录和文档
8. 执行任务并持续记录经验

## 8. 复用规则

1. 优先沿用已验证流程。
2. 对重复出现的任务，优先查历史沉淀。
3. 对可复制的方法，尽量标准化成模板或步骤。
4. 新的稳定经验，及时补进日志和 SOP。

## 9. 更新规则

1. 本文档优先记录“全局共识”。
2. 如果新规则会影响多个项目，先更新本文档。
3. 如果只是单个项目特例，放到对应项目文件夹内部记录。
